# Issue開発スキル

指定されたGitHub Issueに基づいて、ブランチ作成から実装、PR作成まで一連の開発を行う。
スキル内蔵のエージェント定義（`references/` ディレクトリ）を活用し、コード探索・設計・レビューを自動化する。

## 使い方

```
/issue-start #1
/issue-start 1
```

引数からIssue番号を取得する。`#` の有無は問わない。

---

## MCPツール活用方針

プロジェクトに接続済みのMCPサーバーを各Phaseで活用し、情報収集の質を向上させる。

### 利用可能なMCPツール

| MCPサーバー | ツール | 用途 |
|------------|--------|------|
| Brave Search | `brave_web_search` | 公式ジョブガイド・ゲーム仕様・外部技術情報の検索 |
| context7 | `resolve-library-id`, `query-docs` | ライブラリの最新ドキュメント参照 |
| Playwright | `browser_navigate`, `browser_snapshot` 等 | フロントエンド変更時のUI動作確認 |

### フォールバック方針

MCPツールは**補助的な手段**として位置づける。未接続・エラー時は従来の方法（`gh` CLI、手動検索、ユーザーへの確認）にフォールバックし、フロー全体を止めない。

### Phase別MCPツール対応表

| Phase | MCPツール | 活用場面 |
|-------|----------|---------|
| Phase 1（Issue分析） | Brave Search | FF14公式ジョブガイドやゲーム仕様の検索（スキルデータ実装時） |
| Phase 3（コード探索） | context7 | 使用ライブラリのAPIドキュメント参照 |
| Phase 5（実装） | context7 | ライブラリの正確なAPI仕様・最新ドキュメント参照 |
| Phase 6（レビュー） | Playwright | フロントエンド変更時のUI動作確認 |

## Phase 1: Issue分析 & 不足確認

Issueの内容を十分に把握してから実装に入ることが重要。曖昧なまま進めると手戻りが大きくなるため、ここで確実に情報を揃える。

1. `gh issue view <番号> --comments --json title,body,labels,comments,state` でIssueの本文・ラベル・コメントをすべて取得する（`--json` を付けないと GitHub Projects Classic 廃止に伴う GraphQL エラーが発生するため、必ず JSON 形式で取得すること）
2. **マージ済みPRの確認**: `gh pr list --state all --search "<issue番号>" --json number,state,mergedAt,baseRefName` で関連PRの状態を確認する
   - マージ済みPRが存在する場合、ユーザーに通知して追加作業の要否を確認する。不要であれば処理を終了する
   - Issue stateが `OPEN` でもPRがマージ済みのケースがある（developへのマージではGitHubの自動クローズが発動しないため）ので、Issue stateだけで判断しないこと
3. 関連する過去のIssue（本文中のリンクや関連ラベル）があれば参照する
4. **【MCP: Brave Search】** IssueがFF14のスキルデータやゲーム仕様に関わる場合、`brave_web_search` で公式ジョブガイドや関連情報を検索し、実装に必要な数値・仕様を収集する
   - 検索例: `"FF14 竜騎士 ジョブガイド スキル一覧"`, `"FFXIV dragoon job guide skill potency"`
   - MCP未接続時はスキップし、ユーザーに手動確認を依頼する
5. 以下のチェックリストでIssueの記載内容を検証する

### 必須チェックリスト

| 項目 | 確認内容 |
|------|---------|
| 背景・目的 | なぜこの作業が必要か明記されているか |
| 要件 | やること／やらないことが明確か |
| 完了条件 | 何をもって完了とするか定義されているか |
| 技術的な情報 | 実装に必要な設計・定義が十分か |

### 不足時の対応

チェックリストに不足がある場合、**実装に着手せず**ユーザーに聞き返す。

聞き返しのポイント：
- 何が不足しているか具体的に伝える
- 可能であれば選択肢や提案を添える
- ユーザーの回答を得てから次のPhaseに進む

**Example:**
```
Issue #3 の内容を確認しました。以下の点が不足しています：

1. **完了条件が未定義です**
   - 例: 「計算結果が±0.1%以内の誤差であること」のような基準はありますか？

2. **バフの重複適用ルールが不明です**
   - 同種バフは上書き？加算？乗算？
```

## Phase 1.5: Issue自動補完（関連Issue & 完了条件）

実装に入る前に、Issueの「関連Issue」と「完了条件」を自動で補完する。ユーザーがこれらを毎回手書きする負担を減らすことが目的。

### 関連Issueの自動紐づけ

1. `gh issue list --state all --limit 100 --json number,title,labels,state,body` で全Issueを取得する
2. 対象Issueの内容（タイトル・本文・要件）と各Issueを照合し、関連性を判定する
3. 判定基準：
   - 同じ機能領域に関わるIssue（例: バフ関連、タイムライン関連）
   - 依存関係があるIssue（例: 前提となる機能、後続の拡張）
   - 同じコンポーネントを変更するIssue
4. 関連Issueが見つかった場合、Issue本文の「## 関連Issue」セクションを更新する：
   ```bash
   gh issue edit <番号> --body "<更新後の本文>"
   ```

### 完了条件の自動生成

Issueの「やること」セクションの内容が「完了条件」セクションにそのままコピーされているか、完了条件が空・未定義の場合に自動生成する。

1. 「やること」の各項目を基に、検証可能な完了条件を生成する
2. 「やること」をそのまま完了条件にするのではなく、実装・動作確認の観点から条件を具体化する
3. 生成した完了条件でIssue本文の「## 完了条件（Definition of Done）」セクションを更新する

### 補完結果のユーザー確認

自動補完した内容をユーザーに提示し、問題がないか確認してから次のフェーズに進む。

**Example:**
```
Issue #10 の内容を自動補完しました：

**関連Issue:**
- #11 リキャスト概念の追加（同じタイムライン機能に関連）
- #24 バフ・デバフの実装（ダメージ計算の前提機能）

**完了条件（自動生成）:**
- [ ] 〇〇機能が正常に動作すること
- [ ] 既存のテストが通ること

この内容でIssueを更新してよいですか？
```

## Phase 2: ラベル自動付与 & ブランチ作成

### ラベル自動付与

Issueにラベルが付いていない場合、Issue本文の内容から適切なラベルを判定して自動付与する。

判定基準：
- 新しい機能やデータモデルの追加 → `feature`
- 既存機能の不具合修正 → `bug`
- 動作を変えないコード改善 → `refactor`
- ドキュメントのみの変更 → `docs`
- 判断がつかない場合 → ユーザーに確認

```bash
gh issue edit <番号> --add-label "<ラベル名>"
```

### ラベル → ブランチprefix 対応表

| Issueラベル | ブランチprefix |
|-------------|---------------|
| `feature` | `feature/` |
| `bug` | `fix/` |
| `refactor` | `refactor/` |
| `docs` | `docs/` |

### ブランチ作成手順

```bash
git checkout develop
git pull origin develop
git checkout -b <type>/#<issue番号>-<kebab-case説明>
```

## Phase 3: コード探索（code-explorer）

**目的**: 実装に入る前に、既存コードの構造・パターン・関連箇所を深く理解する。ここでの理解が設計と実装の質を左右する。

### 実行手順

1. まず `references/code-explorer-prompt.md`（このスキルの `references/` ディレクトリ内）を読み込む
2. Issueの要件に基づき、**2〜3個のエージェントを並列で起動**する
   - Agent tool で `subagent_type: "Explore"` を指定する
   - `prompt` パラメータの冒頭に、読み込んだ `code-explorer-prompt.md` の内容を「## あなたの役割」として含め、その後に具体的な探索指示を記述する
3. 各エージェントには異なる観点で探索させる：
   - **類似機能の調査**: 「Issue要件に類似する既存機能の実装を包括的にトレースして」
   - **アーキテクチャ把握**: 「変更対象領域のアーキテクチャと抽象化を包括的にマッピングして」
   - **影響範囲の特定**: 「この変更が影響するコンポーネント・テスト・依存関係を特定して」
4. 各エージェントに「重要なファイル5〜10個のリストを含めて」と依頼する
5. エージェントの結果を受け取ったら、**リストアップされたファイルを自分で読んで**理解を深める
6. **【MCP: context7】** 実装で使用するライブラリ（React, Hono, Prisma等）について不明点がある場合、`resolve-library-id` → `query-docs` で最新ドキュメントを参照する
   - 例: Prisma の新しい API を使う必要がある場合、context7 で正確な型定義や使い方を確認する
   - MCP未接続時はスキップし、既存コードのパターンや公式ドキュメントURLをユーザーに共有する
7. 発見した内容をユーザーに簡潔に共有する

### スキップ条件

以下の場合はこのPhaseをスキップして Phase 5（実装）に直行してよい：
- Issueラベルが `bug` で、修正箇所が明確な1ファイル程度の変更
- Issueラベルが `docs` のドキュメント変更
- ユーザーが明示的にスキップを指示した場合

---

## Phase 4: 設計（code-architect）

**目的**: 複数の設計案を比較し、最適なアプローチをユーザーと合意してから実装に入る。事前の設計合意が手戻りを防ぐ。

### 実行手順

1. まず `references/code-architect-prompt.md`（このスキルの `references/` ディレクトリ内）を読み込む
2. Phase 3の探索結果を踏まえ、**2〜3個のエージェントを並列で起動**する
   - Agent tool で `subagent_type: "Explore"` を指定する
   - `prompt` パラメータの冒頭に、読み込んだ `code-architect-prompt.md` の内容を「## あなたの役割」として含め、その後に具体的な設計指示を記述する
3. 各エージェントには異なるアプローチで設計させる：
   - **最小変更**: 「既存コードへの変更を最小限にし、最大限再利用するアプローチで設計して」
   - **クリーン設計**: 「保守性・テスタビリティを重視した理想的なアーキテクチャで設計して」
   - **実用的バランス**: 「実装速度とコード品質のバランスを取るアプローチで設計して」
4. 全アプローチを比較し、**推奨案を理由付きでユーザーに提示**する
5. **ユーザーの承認を得てから**次のPhaseに進む

### 提示フォーマット

```
## 設計案の比較

### 案1: 最小変更
- 変更ファイル: 2個
- メリット: 低リスク、早い
- デメリット: 既存コードとの結合度が高い

### 案2: クリーン設計
- 変更ファイル: 5個（新規2個）
- メリット: テストしやすい、拡張性が高い
- デメリット: 工数が多い

### 案3: 実用的バランス（推奨）
- 変更ファイル: 3個（新規1個）
- メリット: 適度な分離、現実的な工数
- デメリット: 一部結合が残る

→ 案3を推奨します。理由: ...

どのアプローチで進めますか？
```

### スキップ条件

Phase 3と同じ条件に加え：
- 変更が1〜2ファイルで収まる明確な修正の場合
- Issueに具体的な実装方針が既に記載されている場合

---

## Phase 5: 実装

1. CLAUDE.mdのコーディング規約に従って実装する
2. Phase 4で合意した設計方針に沿って実装する（設計フェーズを実行した場合）
3. **【MCP: context7】** ライブラリのAPI仕様が不明な場合、`resolve-library-id` → `query-docs` で最新ドキュメントを参照してから実装する
   - 特にPrisma、Hono、React等のAPI変更が頻繁なライブラリで有効
   - MCP未接続時は既存コードのパターンに倣い、不明点はユーザーに確認する
4. コミットメッセージにIssue番号を含める（例: `feat: スキルデータモデルを追加 #1`）

---

## Phase 6: コードレビュー（code-reviewer）

**目的**: 実装完了後、PRを作成する前にコード品質を確認する。バグや規約違反を早期に検出し、PRの品質を上げる。

### 実行手順

1. まず `references/code-reviewer-prompt.md`（このスキルの `references/` ディレクトリ内）を読み込む
2. 実装が完了したら、**3個のエージェントを並列で起動**する
   - Agent tool で `subagent_type: "Explore"` を指定する
   - `prompt` パラメータの冒頭に、読み込んだ `code-reviewer-prompt.md` の内容を「## あなたの役割」として含め、その後に具体的なレビュー指示を記述する
3. 各エージェントには異なる観点でレビューさせる：
   - **シンプルさ・DRY・可読性**: 「コードの簡潔さ、重複排除、可読性の観点でレビューして」
   - **バグ・機能的正確性**: 「ロジックエラー、エッジケース、バグの観点でレビューして」
   - **プロジェクト規約・アーキテクチャ**: 「CLAUDE.mdの規約遵守と既存アーキテクチャとの整合性をレビューして」
4. 信頼度80以上の指摘のみ集約し、重要度順にユーザーに提示する
5. **【MCP: Playwright】** フロントエンド（React）の変更を含む場合、`browser_navigate` → `browser_snapshot` でUI動作を確認する
   - 開発サーバーが起動済みの場合のみ実行する（起動していない場合はスキップ）
   - MCP未接続時はスキップし、ユーザーに手動確認を依頼する
6. **ユーザーの判断を仰ぐ**（今すぐ修正 / 後で対応 / このまま進める）
7. ユーザーの指示に従って修正を適用する

### スキップ条件

- ユーザーが明示的にスキップを指示した場合
- ドキュメントのみの変更（`docs` ラベル）

---

## Phase 7: PR作成

1. リモートにブランチをプッシュする
2. developブランチへのPRを作成する（CLAUDE.mdのPR規約に従う）:
   ```bash
   gh pr create \
     --base develop \
     --title "<type>: <説明> #<issue番号>" \
     --body "$(cat <<'EOF'
   ## 概要
   ...
   ## 変更点
   ...
   ## テスト
   ...
   closes #<issue番号>
   EOF
   )"
   ```
3. 作成されたPRのURLをユーザーに返す

## Phase 8: Issueへの記録

PR作成後、実装を通じて得られた情報をIssueにコメントとして記録する。後から同じIssueや関連作業を参照したときに、実装の経緯や判断が追えるようにすることが目的。

### 記録する内容

1. **実装メモ**: 実装中に判明した技術情報や設計判断
   ```bash
   gh issue comment <番号> --body "## 実装メモ
   - 採用した技術・ライブラリとその理由
   - 設計上の判断とその根拠
   - ディレクトリ構成やファイル配置の方針"
   ```

2. **ハマりどころ**: トラブルシューティング情報。後から同じ問題に遭遇した際の参考になるよう、**症状・原因・解決策**の3点セットで書く:
   ```bash
   gh issue comment <番号> --body "## ハマりどころ
   **症状**: エラーメッセージや問題の現象
   **原因**: なぜ起きたか
   **解決**: どう直したか"
   ```

3. **その他の補足**: 利用規約上の注意点、外部サービスの制約など、コードだけでは読み取れない情報

### 記録しなくてよいもの

- コードやコミット履歴から明らかに読み取れる内容
- PRの変更点に既に記載済みの内容

---

## 注意事項

- developブランチが存在しない場合は作成する
- 作業中の変更がある場合はユーザーに確認してからブランチを切り替える
- 過去のIssueやPRから関連情報を積極的に収集する
- Phase 3・4・6のエージェント定義は `references/` ディレクトリに配置されている。エージェント起動時は必ずこれらのファイルを読み込み、プロンプトに含めること
- `subagent_type` には組み込みの `"Explore"` を使用する（プラグイン依存を排除するため）
