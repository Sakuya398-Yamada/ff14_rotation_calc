# Phase 1: Issue分析 & 不足確認

Issueの内容を十分に把握してから実装に入ることが重要。曖昧なまま進めると手戻りが大きくなるため、ここで確実に情報を揃える。

## 手順

1. GitHub MCPの `issue_read`（method: `get`）でIssueの本文・ラベルを取得し、`issue_read`（method: `get_comments`）でコメントも取得する
2. **マージ済みPRの確認**: GitHub MCPの `search_pull_requests` で関連PRの状態を確認する（query: `<issue番号>`、owner/repoを指定）
   - マージ済みPRが存在する場合、ユーザーに通知して追加作業の要否を確認する。不要であれば処理を終了する
   - Issue stateが `OPEN` でもPRがマージ済みのケースがあるので、Issue stateだけで判断しないこと
3. 関連する過去のIssue（本文中のリンクや関連ラベル）があれば参照する
4. **【MCP: Brave Search】** IssueがFF14のスキルデータやゲーム仕様に関わる場合、`brave_web_search` で公式ジョブガイドや関連情報を検索し、実装に必要な数値・仕様を収集する
   - 検索例: `"FF14 竜騎士 ジョブガイド スキル一覧"`, `"FFXIV dragoon job guide skill potency"`
   - **利用前の準備**: `ToolSearch query: "select:brave_web_search"` でスキーマをロード
   - **未接続・未設定時のフォールバック**:
     1. `ToolSearch` で `No matching deferred tools found` が返る → `BRAVE_API_KEY` 未設定または stdio 起動失敗。詳細は `.claude/rules/mcp-setup.md` 参照
     2. 組み込みの `WebSearch` ツールで代替検索を試みる（スキーマは `ToolSearch query: "select:WebSearch"` でロード）
     3. それでも必要な情報が得られない場合、実装に必要な数値・仕様をユーザーに直接確認する
   - フロー全体を止めず、情報源を明示した上で次フェーズに進む
5. 以下のチェックリストでIssueの記載内容を検証する

## 必須チェックリスト

| 項目 | 確認内容 |
|------|---------|
| 背景・目的 | なぜこの作業が必要か明記されているか |
| 要件 | やること／やらないことが明確か |
| 完了条件 | 何をもって完了とするか定義されているか |
| 技術的な情報 | 実装に必要な設計・定義が十分か |

## 不足時の対応

チェックリストに不足がある場合、**実装に着手せず**ユーザーに聞き返す。

聞き返しのポイント：
- 何が不足しているか具体的に伝える
- 可能であれば選択肢や提案を添える
- ユーザーの回答を得てから次のPhaseに進む

**Example:**
```
Issue #3 の内容を確認しました。以下の点が不足しています：

1. **完了条件が未定義です**
   - 例: 「計算結果が±0.1%以内の誤差であること」のような基準はありますか？

2. **バフの重複適用ルールが不明です**
   - 同種バフは上書き？加算？乗算？
```

## Phase 1.5: Issue自動補完（関連Issue & 完了条件）

実装に入る前に、Issueの「関連Issue」と「完了条件」を自動で補完する。ユーザーがこれらを毎回手書きする負担を減らすことが目的。

### 関連Issueの自動紐づけ

1. GitHub MCPの `list_issues` で全Issueを取得する（owner/repoを指定、perPage: 100）
2. 対象Issueの内容（タイトル・本文・要件）と各Issueを照合し、関連性を判定する
3. 判定基準：
   - 同じ機能領域に関わるIssue（例: バフ関連、タイムライン関連）
   - 依存関係があるIssue（例: 前提となる機能、後続の拡張）
   - 同じコンポーネントを変更するIssue
4. 関連Issueが見つかった場合、GitHub MCPの `issue_write`（method: `update`）でIssue本文の「## 関連Issue」セクションを更新する

### 完了条件の自動生成

Issueの「やること」セクションの内容が「完了条件」セクションにそのままコピーされているか、完了条件が空・未定義の場合に自動生成する。

1. 「やること」の各項目を基に、検証可能な完了条件を生成する
2. 「やること」をそのまま完了条件にするのではなく、実装・動作確認の観点から条件を具体化する
3. 生成した完了条件でIssue本文の「## 完了条件（Definition of Done）」セクションを更新する

### 補完結果のユーザー確認

自動補完した内容をユーザーに提示し、問題がないか確認してから次のフェーズに進む。

**Example:**
```
Issue #10 の内容を自動補完しました：

**関連Issue:**
- #11 リキャスト概念の追加（同じタイムライン機能に関連）
- #24 バフ・デバフの実装（ダメージ計算の前提機能）

**完了条件（自動生成）:**
- [ ] 〇〇機能が正常に動作すること
- [ ] 既存のテストが通ること

この内容でIssueを更新してよいですか？
```
