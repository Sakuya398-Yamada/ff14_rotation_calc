# Phase 8: Issueへの記録

PR作成後、実装を通じて得られた情報をIssueにコメントとして記録する。後から同じIssueや関連作業を参照したときに、実装の経緯や判断が追えるようにすることが目的。

## 記録する内容

### 1. 実装メモ

実装中に判明した技術情報や設計判断。GitHub MCPの `add_issue_comment` で以下のフォーマットでコメントする：

```
## 実装メモ
- 採用した技術・ライブラリとその理由
- 設計上の判断とその根拠
- ディレクトリ構成やファイル配置の方針
```

### 2. ハマりどころ

トラブルシューティング情報。後から同じ問題に遭遇した際の参考になるよう、**症状・原因・解決策**の3点セットで書く。`add_issue_comment` で：

```
## ハマりどころ
**症状**: エラーメッセージや問題の現象
**原因**: なぜ起きたか
**解決**: どう直したか
```

### 3. その他の補足

利用規約上の注意点、外部サービスの制約など、コードだけでは読み取れない情報。

### 4. 起票した子Issueの記録（Phase 3/5/6 で子Issueを作成した場合）

Phase 3（探索）／Phase 5（実装）／Phase 6（レビュー）でスコープ外問題を子Issueとして起票した場合、親Issueに以下のフォーマットでコメント追記する。後から親Issueを見たときに、このIssue作業中に派生した関連作業が辿れるようにすることが目的。

```
## スコープ外で起票した子Issue

本Issueの作業中に発見したスコープ外の問題を、別Issueとして起票しました：

- #<子Issue番号> <タイトル> (<ラベル>) — 発見Phase: <3/5/6>
- #<子Issue番号> <タイトル> (<ラベル>) — 発見Phase: <3/5/6>
```

- GitHub MCP の `add_issue_comment` で親Issueにコメントする
- 起票した子Issueが複数ある場合は1コメントにまとめる
- 起票が発生しなかった場合はこのセクションを省略してよい

## 記録しなくてよいもの

- コードやコミット履歴から明らかに読み取れる内容
- PRの変更点に既に記載済みの内容
