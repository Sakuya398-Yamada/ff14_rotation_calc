# Phase 3: コード探索（code-explorer）

**目的**: 実装に入る前に、既存コードの構造・パターン・関連箇所を深く理解する。ここでの理解が設計と実装の質を左右する。

## 実行手順

1. Issueの要件に基づき、**`code-explorer` サブエージェントを2〜3個並列で起動**する
   - `Agent` ツールで `subagent_type: "code-explorer"` を指定する
   - エージェント定義は `.claude/agents/code-explorer.md` に集約済み（システムプロンプトの再供給は不要）
2. 各エージェントには異なる観点で探索させる：
   - **類似機能の調査**: 「Issue要件に類似する既存機能の実装を包括的にトレースして」
   - **アーキテクチャ把握**: 「変更対象領域のアーキテクチャと抽象化を包括的にマッピングして」
   - **影響範囲の特定**: 「この変更が影響するコンポーネント・テスト・依存関係を特定して」
3. 各エージェントに「重要なファイル5〜10個のリストを含めて」と依頼する
4. エージェントの結果を受け取ったら、**リストアップされたファイルを自分で読んで**理解を深める
5. **【MCP: context7】** 実装で使用するライブラリ（React, Hono, Prisma等）について不明点がある場合、`resolve-library-id` → `query-docs` で最新ドキュメントを参照する
   - 例: Prisma の新しい API を使う必要がある場合、context7 で正確な型定義や使い方を確認する
   - MCP未接続時はスキップし、既存コードのパターンや公式ドキュメントURLをユーザーに共有する
6. 発見した内容をユーザーに簡潔に共有する
7. **スコープ外問題の一覧化（必須）**: 探索中に当該Issueのスコープ外の問題（別バグ／設計上の違和感／データ誤り等）を見つけた場合、Phase末尾で以下を実施する
   - 起票判定基準（`SKILL.md` 「スコープ外問題の起票提案」節）に従って候補を絞り込む
   - GitHub MCP の `search_issues` で既存Issueの重複を確認する
   - 候補ごとに SKILL.md の起票フォーマットで提示し、Y/E/N の選択をユーザーに確認する
   - 承認された候補は `issue_write`（method: `create`）で子Issueとして起票する（本文末尾に `親Issue: #<親番号>` を明記）
   - 見つからなかった場合は「スコープ外問題の検出: なし」と明示的に報告する

## 並列起動の例

```
Agent(subagent_type: "code-explorer", prompt: "src/server/services/ 配下にあるダメージ計算系サービスの実装を全部トレースして。エントリポイント・データフロー・依存関係をまとめ、関連ファイルを5〜10個リストアップして。")
Agent(subagent_type: "code-explorer", prompt: "src/client/components/ 配下のスキル選択UIアーキテクチャを把握して。状態管理・APIコール・データ変換を中心にまとめ、関連ファイルを5〜10個リストアップして。")
```

## スキップ条件

以下の場合はこのPhaseをスキップして Phase 5（実装）に直行してよい：
- Issueラベルが `bug` で、修正箇所が明確な1ファイル程度の変更
- Issueラベルが `docs` のドキュメント変更
- ユーザーが明示的にスキップを指示した場合
