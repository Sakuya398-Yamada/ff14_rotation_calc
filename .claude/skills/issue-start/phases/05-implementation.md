# Phase 5: 実装

1. `CLAUDE.md` および `.claude/rules/coding-standards.md` のコーディング規約に従って実装する
2. Phase 4で合意した設計方針に沿って実装する（設計フェーズを実行した場合）
3. **【MCP: context7】** ライブラリのAPI仕様が不明な場合、`resolve-library-id` → `query-docs` で最新ドキュメントを参照してから実装する
   - 特に Prisma、Hono、React 等の API変更が頻繁なライブラリで有効
   - MCP未接続時は既存コードのパターンに倣い、不明点はユーザーに確認する
4. **中間報告（必須）**: 大きなファイル（概ね 300 行以上）や複数ファイルを一度に読み込んだ直後は、長時間の内部推論を始める前に以下を 2〜4 行でユーザーに報告する
   - 把握したこと（既存構造・関連箇所）
   - これから行う変更の計画
   - 迷いがある点があればその時点で質問する（推測で先に進まない）
5. コミットメッセージは `<type>: <subject> #<issue番号>` 形式で、Issue番号を必ず含める
   - 詳細は `.claude/rules/git-conventions.md`
   - **PreToolUse hook が `git commit` を検証する**ため、規約違反はブロックされる

## コミット粒度

- 論理的に独立した変更単位ごとにコミットする
- 1コミット = 1テーマを目安に、レビューしやすく分割する
- WIPコミットは避け、各コミットがビルド可能・テスト通過する状態を保つ

## スコープ外問題の記録と末尾提案

実装中に当該Issueのスコープ外の問題（周辺コードのバグ／仕様乖離／技術負債等）に遭遇した場合は **その場で直さず、メモとして記録するだけに留める**。1 Issue = 1 PR 原則を守り、実装フローを分断しないため。

1. **即時対応しない**: 当該Issueの完了条件（DoD）に含まれない変更は、ブランチ上で修正しない
2. **メモに記録する**: 発見時点で以下を控えておく
   - 何が問題か（1〜3行）
   - 該当ファイルと行番号（例: `src/client/data/pct-skills.ts:L481-498`）
   - バグの場合は再現手順
3. **Phase 5 末尾でまとめて提案**: 実装＋コミットが一通り完了したタイミングで、記録した問題を一括でユーザーに提示する
   - 起票判定基準は `SKILL.md` 「スコープ外問題の起票提案」節を参照
   - GitHub MCP の `search_issues` で既存Issueの重複を確認してから提案する
   - 候補ごとに Y/E/N の選択をユーザーに確認し、承認されたものだけ `issue_write`（method: `create`）で起票する
   - 本文末尾に `親Issue: #<親番号>` を明記する
4. **発見が無かった場合**: 「スコープ外問題の検出: なし」と明示的に報告する

> 例外: 実装の阻害になる問題（進行不能な致命バグ）に遭遇した場合は、即座にユーザーに報告し、当該Issueのスコープを広げるか・別Issueに切り出して先に対応するかを判断する。
