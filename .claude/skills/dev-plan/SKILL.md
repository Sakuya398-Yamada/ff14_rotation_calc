# 開発計画スキル

指定されたIssueを分析し、依存関係に基づいた実装順序（Wave構造）を策定する。

## 使い方

```
/dev-plan #1 #2 #3
/dev-plan #10-15
```

引数からIssue番号を取得する。範囲指定（`#10-15`）も対応する。

## 実行手順

### Step 1: Issue取得

指定されたIssueを `gh issue view <番号> --comments` ですべて取得し、内容を把握する。

### Step 2: 依存関係分析

Issue間の依存関係を以下の基準で判定する。データの流れや参照関係を読み取ることが重要で、明示的に「依存」と書かれていなくても、あるIssueの成果物を別のIssueが前提としていれば依存関係として扱う。

| パターン | 配置 |
|---------|------|
| データモデル・スキーマ定義 | 最初のWave |
| 共通ユーティリティ・基盤コード | 早いWave |
| 独立した機能（互いに参照なし） | 同一Waveで並列化 |
| 他のIssueの成果物を使う機能 | 後のWave |
| UIコンポーネント（データ層に依存） | 後のWave |

### Step 3: Wave分類 & 出力

依存関係に基づきWaveに分類し、以下のフォーマットで出力する。

```markdown
# 開発計画

## Wave 1（直列 - 基盤）
- #1: スキルデータモデルの作成
  - 理由: 他の全機能がこのモデルに依存

## Wave 2（並列可能）
- #2: ダメージ計算エンジン
  - 依存: #1
- #3: バフ・デバフ管理
  - 依存: #1

## Wave 3（並列可能）
- #4: ローテーション入力UI
  - 依存: #2, #3

---

## 実装指示

### Issue #1: スキルデータモデルの作成
**ブランチ**: `feature/#1-add-skill-data-model`
**概要**: ...
**実装ポイント**:
- ...
```

## 注意事項

- 各WaveはGitHub Flow準拠で、前のWaveのPRがマージされてから着手する
- 同一Wave内のIssueは並列開発可能（worktree活用）
- Waveが5以上になる場合はIssueの粒度を見直すことを提案する
